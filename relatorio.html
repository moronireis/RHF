Campos Capturados — Webhook Meta
Estrutura armazenada em rhf_messages
phone
string
Número do remetente (wa_id extraído de message.from)
Ex: "5511999998888"
direction
string
Detecção automática: mensagens com prefixo *NomeAgente:* → outbound (via ChatGuru sign), demais → inbound
Ex: "inbound" | "outbound"
content
string
Texto da mensagem. Para mídia: "[Imagem]", "[Audio]", "[Video]", "[Documento]"
Ex: "Olá, gostaria de me candidatar"
message_type
string
Tipo mapeado: text → chat, image, audio → ptt, video/document → file, sticker → image
Ex: "chat" | "image" | "ptt" | "file"
chatguru_message_id
string
WAMID (WhatsApp Message ID) extraído de message.id
Ex: "wamid.ABCDEfgh..."
chatguru_chat_id
string
Mesmo que phone (identificador da conversa no modelo ChatGuru)
Ex: "5511999998888"
raw_webhook
jsonb
Metadados extras do webhook:
{ source: "meta_cloud_api", contact_name: "João Silva", // de contacts[].profile.name wamid: "wamid.ABCDEfgh...", // id original type: "text" // tipo bruto do Meta }

Payload do Webhook ChatGuru
Campos recebidos a cada evento de mensagem
nome
string
Nome do contato no ChatGuru
Ex: "Ana Paula Santos"
celular
string
Número de telefone (qualquer formato) — normalizado com replace(/\D/g, '')
Ex: "(11) 9 9999-8888" → "11999998888"
texto_mensagem
string
Conteúdo textual da mensagem
Ex: "Tenho interesse na vaga de analista"
tipo_mensagem
string
Tipo do conteúdo enviado. Default: "chat"
Ex: "chat" | "image" | "audio" | "video" | "file"
phone_id
string
ID do número WhatsApp Business no ChatGuru
Ex: "ph_abc123"
chat_id
string
Identificador único da conversa no ChatGuru. Salvo em chatguru_chat_id
Ex: "ch_xyz789"
email
string
Email do candidato (quando disponível no cadastro ChatGuru)
Ex: "candidato@email.com"
url_arquivo
string
URL do arquivo de mídia (imagens, documentos, áudios). Armazenado como content quando texto_mensagem está vazio
Ex: "https://cdn.chatguru.com.br/files/abc.pdf"
Fluxo de Processamento
O que acontece quando o webhook é recebido
1
Recebe POST do ChatGuru
Extrai: nome, celular, texto_mensagem, tipo_mensagem, chat_id, email
2
Upsert em candidates
SELECT por phone — se existir, usa o id existente. Se não, INSERT com name + phone + chatguru_chat_id + status "new"
3
INSERT em rhf_messages
direction sempre "inbound", content = texto_mensagem ou url_arquivo, raw_webhook = payload completo
4
INSERT em sync_log
source = "chatguru", action = "webhook_received", status = "success". Retorna 200 em caso de erro também (evita retries do ChatGuru)

Webhook — Evento Configurado
Candidato mudou de estágio em uma vaga
Pandapé dispara este evento quando o recrutador move um candidato de pasta
Aguardando config.

URL a configurar no painel Pandapé:

https://rhf-proposta.vercel.app/api/webhook/pandape

Payload recebido:

IdMatch
number
ID único do match (candidato ↔ vaga) no Pandapé. Usado para buscar dados completos via GET /v2/matches/{IdMatch}
Ex: 602649148
IdVacancy
number
ID da vaga no Pandapé. Usado para buscar título e detalhes via GET /v2/vacancies/{IdVacancy}
Ex: 2656804
IdVacancyFolderFrom
number
ID da pasta de origem (estágio anterior no funil de seleção)
Ex: 17098398
IdVacancyFolderTo
number
ID da pasta de destino (novo estágio do candidato)
Ex: 17098395
EventDate
string
Timestamp ISO 8601 do momento do evento no Pandapé
Ex: "2025-12-18T14:36:24.3750583"
Endpoints REST Pandapé
GET
/v2/matches/{idMatch}
Retorna dados completos do candidato na vaga: CandidateName, CandidateEmail, CandidatePhone, estágio atual, histórico de movimentações, dados do processo seletivo.
GET
/v2/vacancies/{idVacancy}
Retorna detalhes da vaga: Title, departamento, localização, requisitos, status da vaga, datas de abertura/encerramento.
POST
/v2/vacancies
Cria nova vaga — disponível para Fase 2 (envio de vagas RHF para Pandapé).
POST
/v2/requests
Cria requisição de vaga — disponível para Fase 2.

Obter token (client_credentials) POST https://api.pandape.com.br/oauth/token Content-Type: application/x-www-form-urlencoded grant_type=client_credentials client_id={PANDAPE_CLIENT_ID} client_secret={PANDAPE_CLIENT_SECRET} // Resposta { "access_token": "eyJhbGciOiJSUzI1...", "token_type": "Bearer", "expires_in": 3600 } // 2. Usar token em todas as requisições Authorization: Bearer {access_token} // Token cache: lib/pandape.js faz cache com buffer de 60s antes do vencimento
